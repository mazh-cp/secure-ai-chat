# Release Freeze — v1.0.12

**Stable scope:** File upload → registry → parse → scan (Lakera optional) → embed/index → retrieve → chat inject → output scan → citations.

No further feature changes to this pipeline after release. Patches only for security and critical bugs.

---

## Release Freeze Checklist

### Environment variables (required / optional)

| Variable                           | Required                 | Notes                                                         |
| ---------------------------------- | ------------------------ | ------------------------------------------------------------- |
| `NODE_ENV`                         | No                       | Set to `production` for prod; defaults safe for dev           |
| `PORT`                             | No                       | Default `3000`                                                |
| `HOSTNAME`                         | No                       | Default `0.0.0.0`                                             |
| `REGISTRY_DB_PATH`                 | No (recommended in prod) | SQLite registry; absolute path recommended for multi-process  |
| `UPLOADS_DIR`                      | No (recommended in prod) | Uploaded file bytes; absolute path recommended                |
| `DATA_DIR`                         | No                       | Parent for default `app.db` and uploads when paths unset      |
| `OPENAI_API_KEY`                   | Yes for chat             | Or set via Settings UI                                        |
| `LAKERA_ENABLED`                   | No                       | Inferred from `LAKERA_API_KEY` / `LAKERA_AI_KEY`              |
| `LAKERA_API_KEY` / `LAKERA_AI_KEY` | If using Lakera          | Guard at upload + retrieval + output                          |
| `LAKERA_PROJECT_ID`                | No                       | Optional project-scoped key                                   |
| `LAKERA_FAIL_CLOSED`               | No                       | If set, treat Lakera errors as block (documented in README)   |
| `LAKERA_TIMEOUT_MS`                | No                       | Request timeout for Lakera (documented in README)             |
| `BASE_URL`                         | No                       | Used by smoke tests / callbacks; e.g. `http://localhost:3000` |
| `STORAGE_DIR`                      | No                       | Default `./.storage`                                          |

See `.env.example` and [README Production section](../README.md#environment-variables-for-production).

### Known good Node version

- **Node.js:** `24.13.0` (LTS) — pinned in `package.json` `engines` and `.nvmrc` (if present).
- Check: `node -v` → `v24.13.0`

### Build and smoke commands

```bash
# Install (frozen lockfile)
npm ci

# Lint and typecheck
npm run lint
npm run typecheck

# Production build
npm run build

# Start production server
npm run start

# Smoke tests (server must be running for RAG smoke)
BASE_URL=http://localhost:3000 npm run smoke:rag-csv
BASE_URL=http://localhost:3000 npm run smoke:rag-csv-multifile
# Optional (requires LibreOffice + .numbers file):
# BASE_URL=http://localhost:3000 npm run smoke:rag-numbers
```

### Regression tests to run before release

1. **Health:** `curl -s http://localhost:3000/api/health` → 200, JSON with `status`, `version`
2. **Files list:** `curl -s -b <cookies> http://localhost:3000/api/files/list` → 200
3. **RAG status:** `curl -s -b <cookies> http://localhost:3000/api/rag/status` → 200, consistent counts after upload
4. **Smoke:** `BASE_URL=http://localhost:3000 npm run smoke:rag-csv` — must pass (cookie jar used for same owner)
5. **With Lakera disabled:** All above pass with `LAKERA_ENABLED=false` or no Lakera keys
6. **With Lakera enabled:** Benign CSV passes; malicious injection content quarantined/dropped
7. **Secrets:** No secrets in repo; `npm run check:secrets` (or gitleaks) passes

### Dev-only behavior (disabled in production)

- **Debug routes:** `GET /api/debug/*` return 404 when `NODE_ENV === 'production'`.
- **Logging:** No chunk previews or raw file contents in responses or production logs; only safe metadata (e.g. file_id, owner_id, counts) in logs.

## Version and tag

### Create and push tag (maintainer)

```bash
git tag v1.0.12
git push origin v1.0.12
```

### Upgrade from tag (deployer)

```bash
cd /opt/secure-ai-chat
git fetch origin tag v1.0.12
git checkout v1.0.12
npm ci
npm run build
sudo systemctl restart secure-ai-chat
# Verify:
curl -s http://localhost:3000/api/health
curl -s http://localhost:3000/api/rag/status
```

Or run: `./scripts/upgrade-to-1.0.12.sh`
